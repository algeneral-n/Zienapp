# ZIEN Platform Constitution
## القواعد المطلقة والمبادئ الحاكمة للمنصة
**الإصدار:** 1.0  
**التاريخ:** 2026-02-25  
**الحالة:** وثيقة دائمة -- لا تُعدَّل إلا بقرار صريح من الفاوندر

---

## 1) أسلوب الكتابة والهوية البصرية العامة

### 1.1 قواعد الكتابة

- ممنوع استخدام الإيموجي في أي جزء من المنصة أو الوثائق أو النصوص أو الردود.
- ممنوع البهرجة البصرية في النصوص:
  - لا استخدام غير منضبط للأحرف الكبيرة.
  - لا تنسيقات عشوائية.
  - لا ألوان كثيرة أو غير متسقة.
- أسلوب الكتابة يجب أن يكون:
  - واضحًا
  - رسميًا
  - مباشرًا
  - متسقًا عبر الويب والموبايل

### 1.2 قواعد الـ UI

- لون رئيسي واحد فقط للمنصة.
- لون ثانوي خفيف يستخدم للبطاقات والأزرار والعناصر المساندة.
- ألوان منفصلة وواضحة فقط لـ:
  - النجاح (Success)
  - الخطأ (Error)
  - التحذير (Warning) عند الحاجة
- أي عنصر يحتاج تمييزًا:
  - يتم تمييزه عبر بوردر خفيف أو تغيير بسيط في الخلفية.
  - يمنع استخدام ألوان صاخبة أو مؤثرات مبالغ فيها.
  - يمنع الاعتماد على gradients القوية أو الزخرفة غير العملية.

## 2) الترجمة متعددة اللغات (i18n) -- 15 لغة بدون تنازل

### 2.1 مبدأ الترجمة الكامل

- المنصة تدعم 15 لغة بشكل احترافي.
- عند اختيار لغة:
  - كل النصوص تظهر بنفس اللغة.
  - لا يسمح بخلط لغتين في نفس الشاشة أو نفس السياق.

### 2.2 قواعد التنفيذ

- كل النصوص في الواجهة يجب أن تكون عبر مفاتيح ترجمة فقط.
- يمنع استخدام نصوص hard-coded داخل الكود.
- يتم اعتماد:
  - ملف ترجمة مركزي موحّد.
  - هيكل مفاتيح ثابت ومنظم (حسب الصفحات والموديولات).
- أي صفحة أو موديول جديد لا يُقبل إلا إذا كان:
  - مترجمًا بالكامل.
  - ومتوافقًا مع النظام المركزي للترجمة.

### 2.3 نطاق الترجمة

يشمل الترجمة بالكامل:

- الصفحات العامة
- صفحات الدخول والتسجيل
- لوحات التحكم
- النماذج والحقول
- رسائل الخطأ والنجاح
- الإشعارات
- محتوى الـ AI
- محتوى التقارير
- محتوى الأكاديمي والأسئلة الشائعة والسياسات

## 3) RARE AI -- القلب الذكي للمنصة

### 3.1 زر عائم دائم

- يوجد زر عائم لـ RARE AI في كل صفحة (Web + Mobile).
- الزر متاح دائمًا مع اختلاف السلوك حسب:
  - هل المستخدم ضيف أم مسجل دخول.
  - دور المستخدم.
  - الصفحة الحالية.
  - الشركة النشطة.

### 3.2 الوعي بالسياق

RARE AI يجب أن يكون واعيًا بـ:

- الصفحة الحالية.
- الشركة الحالية.
- الدور الحالي.
- الموديولات المفعلة.
- الصلاحيات المسموحة.
- لغة المستخدم المختارة.

### 3.3 هيكلة الـ Agents

يجب وجود Agents متخصصة على 3 مستويات:

**أ) Agent حسب القسم**

- HR Agent
- Accounting Agent
- CRM Agent
- Logistics Agent
- Projects Agent
- Store/POS Agent
- Legal/Compliance Agent
- Support Agent

**ب) Agent حسب الخدمة**

- Payroll Agent
- Invoicing Agent
- Tax Agent
- Attendance Agent
- Fleet/GPS Agent
- Meeting Summary Agent
- Recruitment Agent
- Inventory Agent

**ج) Agent حسب الدور**

- GM Agent
- Assistant GM Agent
- Department Manager Agent
- Supervisor Agent
- Accountant Agent
- HR Officer Agent
- Driver Agent
- Field Employee Agent
- Client Portal Agent

### 3.4 قواعد تنفيذ الأوامر

- RARE AI لا ينفذ أوامر خارج صلاحية الدور.
- أي أمر تنفيذي يجب أن يمر عبر:
  - التحقق من الدور.
  - التحقق من صلاحية الموديول.
  - التحقق من بيانات الشركة.
  - التحقق من حدود العملية.
- أمثلة:
  - موظف عادي لا يمكنه طلب تقرير رواتب كامل.
  - موظف لا يمكنه استخراج مستندات المدير.
  - تفعيل Integration مدفوع لا يتم إلا من مخول.

### 3.5 مستويات أوامر RARE (سياسة رسمية)

تقسم أوامر RARE إلى فئات:

**1) Read-Only**
- قراءة وعرض معلومات مسموح بها فقط.
- لا تغيّر البيانات.

**2) Suggest-Only**
- اقتراح تحسينات وخطوات.
- لا تنفيذ مباشر.

**3) Execute-With-Permission**
- تنفيذ عمليات مسموح بها للدور الحالي.
- تسجل في Audit Log.

**4) Sensitive Actions**
- عمليات حساسة (حذف، تعطيل، شراء، تعديل صلاحيات).
- تحتاج:
  - صلاحية أعلى.
  - وقد تحتاج موافقة إضافية (مثل GM).

### 3.6 صفحة مركزية لـ RARE AI

يجب وجود صفحة كاملة لـ RARE تشمل:

- مرجع قدرات RARE.
- سجل الأوامر الحساسة (Audit).
- حالة الـ Agents.
- سياسات الاستخدام.
- ضوابط التنفيذ.
- مراقبة إساءة الاستخدام ومحاولات الاستغلال.

## 4) العزل التام Multi-Tenant + Role Isolation

### 4.1 قاعدة العزل الأساسية

- لا تسريب بيانات بين أي شركتين (Tenants) تحت أي ظرف.
- العزل يجب أن يكون فعليًا في:
  - قاعدة البيانات.
  - سياسات RLS.
  - الكاش.
  - استدعاءات الـ AI.
  - التقارير.
  - البحث الداخلي.

### 4.2 العزل داخل الشركة نفسها

- كل مستخدم يرى فقط ما يناسب:
  - دوره.
  - قسمه.
  - الموديولات المفعلة له.
  - الصلاحيات الفعلية.
- لا يكفي إخفاء الواجهة فقط؛ يجب منع الوصول من المصدر (DB/API/RLS).

### 4.3 قاعدة company_id

- كل جدول أعمال (business table) يجب أن يحتوي على company_id.
- كل استعلام يجب أن يكون مقيدًا بـ company_id.
- سياسات RLS تفرض ذلك إجباريًا ولا تعتمد على الواجهة فقط.

### 4.4 عزل الـ AI Context

- ممنوع تمرير سياق من شركة إلى أخرى للـ AI.
- أي Prompt أو Context يُبنى فقط من بيانات الشركة الحالية والدور الحالي.
- يتم تسجيل أي استخدام حساس للـ AI.

## 5) التسعير الذكي -- ناتج من البروفيجنينج وليس صفحة ثابتة

### 5.1 إلغاء صفحة Pricing الثابتة

- لا توجد Pricing Page ثابتة بأسعار عامة.
- التسعير النهائي يظهر بعد تحليل متطلبات الشركة أثناء البروفيجنينج.

### 5.2 نموذج التسعير الديناميكي

السعر النهائي يحسب من:

- نوع الشركة / النشاط.
- حجم الشركة.
- عدد المستخدمين (Seats).
- الموديولات المطلوبة.
- الـ Integrations المطلوبة.
- مستوى استخدام AI.
- خطط التخزين / التتبع / الاجتماعات / الدفع.

### 5.3 صيغة التسعير (مبدئيًا)

Dynamic Pricing = Base Fee + Modules + Seats + Integrations + Usage + Support Tier

### 5.4 وقت إظهار السعر

- السعر يظهر في نهاية رحلة التسجيل/البروفيجنينج.
- ثم تعرض خيارات التفعيل والـ Integrations المناسبة.

## 6) محرك البروفيجنينج -- المصنع الأساسي للمنصة

### 6.1 الهدف

محرك البروفيجنينج مسؤول عن:

- تحليل بيانات التسجيل الأولية للشركة.
- اختيار Blueprint مناسب.
- بناء موديولات الشركة تلقائيًا.
- تطبيق بيانات التهيئة (Seed Packs).
- إعداد بيئة تشغيل جاهزة.
- توليد السعر الديناميكي.
- اقتراح الـ Integrations المناسبة.

### 6.2 أنواع الأنشطة المستهدفة (أمثلة أساسية)

المحرك يجب أن يدعم قطاعات كثيرة، منها:

- السوبرماركت والتجزئة
- المصانع والتصنيع
- المولات
- المهن والفنيين
- الجمعيات التعاونية والخيرية
- العقارات والتطوير العقاري
- التخليص الجمركي والاستيراد/التصدير
- الاستشارات (هندسية/فنية/خدمية)
- الإنتاج الإعلامي والتصوير
- المقاولات والإنشاءات
- البيع المباشر والدليفري
- الخدمات اللوجستية
- الشركات الخدمية والإدارية

### 6.3 مدخلات البروفيجنينج

تسجيل الشركة يجب أن يجمع على الأقل:

- نوع النشاط.
- النشاط الفرعي.
- الدولة / المنطقة.
- حجم الشركة.
- عدد الموظفين.
- عدد الفروع.
- الخدمات المطلوبة.
- احتياجات التشغيل (POS / مخزون / توصيل / مشاريع / HR / CRM...).
- بيانات المدير العام.
- بيانات الفوترة الأولية.

### 6.4 مخرجات البروفيجنينج

المحرك يجب أن ينتج:

- شركة مهيأة (Company).
- الموديولات المفعلة.
- الأقسام الافتراضية.
- الأدوار الافتراضية.
- Seed Data (ضرائب، قوالب فواتير، شجرة حسابات، سياسات أولية).
- Job Status + Logs.
- Dynamic Pricing.
- قائمة Integrations مقترحة جاهزة للتفعيل.

### 6.5 التخصيص بعد البروفيجنينج

بعد اكتمال التهيئة:

- المدير العام (أو المخول) يستطيع:
  - إضافة موديولات.
  - حذف موديولات.
  - تعديل إعدادات الموديولات.
  - تعديل الأقسام.
  - تعديل الصلاحيات.
- أي تعديل كبير يعيد تقييم السعر تلقائيًا.

### 6.6 صفحة Integrations بعد البروفيجنينج

بعد انتهاء التهيئة مباشرة، تظهر صفحة Integrations ذكية تعرض:

- ما هو مطلوب لتشغيل النشاط.
- ما هو اختياري.
- ما يحتاج رسوم إضافية.
- ما يمكن تفعيله لاحقًا.

## 7) صفحة الانتيجريشن الموحدة -- Public + Role-Aware + Paid Add-ons

### 7.1 طبيعة الصفحة

- صفحة Integrations تظهر بشكل عام (Public Preview).
- لكن التفعيل الحقيقي والدفع:
  - لا يتمان إلا بعد تسجيل الدخول.
  - وبصلاحية مناسبة.

### 7.2 من يحق له التفعيل

- المدير العام (GM).
- مساعد المدير العام (إذا مفوض).
- قسم الحسابات (حسب نوع الـ Integration).
- مدير القسم المختص (في بعض الحالات).
- غير ذلك: عرض فقط دون تفعيل.

### 7.3 أقسام الـ Integrations المطلوبة

**أ) أنظمة الدفع**
- Stripe
- Apple Pay (عبر مزود مناسب)
- Google Pay (عبر مزود مناسب)
- مزودات محلية/إقليمية (مثل مزودات الإمارات)
- بوابات دفع أخرى حسب الدولة

**ب) التسويق الذكي**
- Google Ads
- YouTube
- Meta (Facebook / Instagram)
- WhatsApp
- Email Marketing
- حملات متعددة القنوات

**ج) التخزين السحابي**
- Supabase Storage
- Cloudflare R2
- S3-compatible providers
- أرشفة الملفات والسجلات

**د) الاتصالات والاجتماعات**
- Vonage (فيديو/صوت)
- Chat integrations
- تلخيص الاجتماعات بالذكاء الاصطناعي

**هـ) الخرائط والتتبع**
- GPS tracking
- Maps
- تتبع السائقين والمندوبين
- Geofencing / Route optimization

**و) الأتمتة والذكاء**
- توليد العقود والفواتير
- أوامر صوتية
- سير عمل آلي
- مساعدات ذكية متقدمة

### 7.4 قواعد التسعير الخاصة بالـ Integrations

لكل Integration يجب دعم:

- رسوم تفعيل (Setup Fee) إن وجدت.
- رسوم شهرية (Subscription).
- رسوم استخدام (Usage-based).
- حالة الدفع والتجديد.
- فشل الدفع وإعادة المحاولة.
- Webhooks للتحديث التلقائي.

## 8) الصفحات العامة Public Pages -- متطلبات إلزامية

### 8.1 Landing Page

يجب أن تكون:
- دعائية قوية ومصقولة.
- واضحة في عرض قيمة المنصة.
- متوازنة بصريًا.
- مناسبة لكل اللغات (بما فيها RTL).

عناصر إلزامية:
- زر "سجل الآن" في الهيدر.
- زر "سجل الآن" في الـ Hero.
- زر Login في الهيدر.
- اختيار اللغة.
- اختيار الثيم.
- عرض الميزات الأساسية.
- روابط أساسية في الفوتر.
- زر RARE العائم دائمًا.

### 8.2 Register Page (تسجيل شركة)

- هذه ليست صفحة إنشاء مستخدم عادي.
- هي بوابة بدء تسجيل شركة.
- تقود مباشرة إلى Workflow البروفيجنينج.
- تجمع كل مدخلات التهيئة الأولية.
- تنتهي بعرض:
  - السعر الديناميكي.
  - الـ Integrations المقترحة.
  - خيارات التفعيل.

### 8.3 Login Page

طرق الدخول المدعومة:
- Email + Password
- Phone OTP
- Google
- Apple

قاعدة صارمة:
- وسيلة الدخول يجب أن تطابق هوية مستخدم مسجل بالفعل داخل الشركة.
- لا يسمح للمستخدم أن يرى إلا:
  - محتوى شركته.
  - ما يسمح به دوره.
  - ما هو مفعل من الموديولات.

### 8.4 Academy Page

- مكتبة تعليمية كاملة لكل أجزاء المنصة.
- دروس + مسارات تدريب.
- اختبارات.
- شهادات إتمام داخلية.
- RARE يمكنه اقتراح أو إنشاء خطة تدريب حسب الدور.

### 8.5 FAQ + Keep in Touch

- صفحة FAQ ديناميكية.
- تسمح باستقبال أسئلة الجمهور.
- الأسئلة المهمة تتحول إلى FAQ entries بعد مراجعة.
- يمكن دعم التصنيف والبحث.

### 8.6 Contact Page

تشمل:
- Contact Form (شكاوى / اقتراحات / طلبات / استفسارات).
- زر اتصال مباشر.
- زر واتساب.
- بريد إلكتروني.
- خريطة وموقع (إذا متاح).

### 8.7 Legal Pages

تشمل على الأقل:
- Privacy Policy
- Terms of Service
- Usage Policies
- User Rights
- Refund Policy (إن وجدت)
- Security and Compliance

## 9) الأدوار والدرجات الوظيفية -- نموذج حقيقي لسوق العمل

### 9.1 مبدأ عام

- الأدوار ليست شكلية.
- كل دور يؤثر على:
  - الصفحات التي تظهر.
  - العمليات المسموحة.
  - بيانات التقارير.
  - صلاحيات الـ AI.
  - صلاحيات الـ Integrations.

### 9.2 الأدوار الإدارية العليا

- المدير العام (GM)
- مساعد المدير العام (بتفويض واضح)
- مدير قسم
- مشرف

### 9.3 الأدوار التشغيلية

- موظف أول
- موظف عادي
- موظف جديد
- متدرب

### 9.4 أدوار خاصة حسب النشاط

- سائق
- مندوب
- موظف ميداني
- كاشير
- موظف مخزن
- محاسب
- مسؤول موارد بشرية
- موظف خدمة عملاء

### 9.5 أدوار خارجية

- عميل (Client Portal User)
- مورد (عند الحاجة)
- شريك (عند الحاجة)

### 9.6 قواعد الصلاحيات

كل دور يجب أن يعرّف بدقة في:
- الموديولات المتاحة له.
- نوع العمليات (قراءة/إنشاء/تعديل/اعتماد/حذف).
- حدود البيانات (قسمه فقط / كل الشركة / فروع معينة).
- صلاحيات الـ AI.
- صلاحيات الـ Integrations.

## 10) الخدمات الأساسية الدائمة (Core Modules)

هذه الخدمات يجب أن تكون قوية دائمًا، سواء فُعّلت تلقائيًا في البروفيجنينج أو أضيفت لاحقًا.

### 10.1 المحاسبة والمالية

يشمل:
- الفواتير.
- المدفوعات.
- الإيرادات.
- المصروفات.
- الرواتب والسلف.
- الاشتراكات.
- الضرائب.
- قوالب الفواتير القابلة للتخصيص.
- تقارير مالية.
- Agent محاسبة ذكي.

### 10.2 الموارد البشرية (HR)

يشمل:
- ملفات الموظفين.
- العقود.
- الحضور والانصراف.
- الإجازات والغيابات.
- التأمينات.
- إصابات العمل.
- التوظيف واستقبال السير الذاتية.
- التدريب والاختبارات الداخلية.
- Agent HR ذكي لكل دور.

### 10.3 CRM + المبيعات + Client Portal

يشمل:
- العملاء المحتملين.
- الفرص البيعية.
- عروض الأسعار.
- العقود.
- الفواتير والإيصالات.
- متابعة التواصل.
- Portal للعملاء.
- Agent مبيعات/خدمة عملاء.

### 10.4 بوابة الموظفين

تنقسم إلى:
- موظفون داخليون.
- موظفون ميدانيون/خارجيون.

وتشمل:
- حضور/انصراف ذكي.
- استلام المهام.
- رفع التقارير.
- أرشيف المستندات.
- ربط GPS للسائقين والمندوبين.

### 10.5 Global Store + POS

يشمل:
- المنتجات.
- المخزون.
- نقاط البيع.
- خدمة العملاء.
- قنوات البيع المباشر.
- بوابة بيع إلكتروني (حسب الحاجة).
- Agent مخزون/مبيعات.

### 10.6 إدارة المشاريع والمهام

يشمل دورة العمل بين الأقسام:
- من المبيعات.
- إلى الحسابات للتسعير.
- إلى إدارة المشاريع للتخطيط.
- إلى التشغيل/الميدان للتنفيذ.
- مع التتبع الكامل للحالة والتقدم.

### 10.7 الشات الجماعي + Presence

يشمل:
- محادثات قسمية / جماعية.
- حالة الموظف:
  - Online
  - Offline
  - Away
  - On Leave
- ظهور الحالة يكون حسب الصلاحية:
  - كاملة لـ HR وGM.
  - جزئية للأقسام المعنية فقط.

## 11) Founder / Control Plane -- لوحة تحكم المنصة نفسها

### 11.1 الهدف

لوحة الفاوندر ليست لوحة شركة؛ هي لوحة إدارة المنصة ZIEN نفسها.

### 11.2 المهام الأساسية

- مراقبة صحة المنصة.
- مراقبة الأمن والتنبيهات.
- إدارة الـ blueprints الافتراضية.
- إدارة الـ modules catalog.
- إدارة الـ integrations catalog.
- مراقبة البروفيجنينج jobs.
- مراقبة الاشتراكات والفوترة العامة.
- متابعة الشكاوى والطلبات.
- تقارير يومية/أسبوعية/شهرية.
- أوتوميشن إداري للمنصة نفسها.

### 11.3 حدود صلاحيات الفاوندر

- الفاوندر لا يرى البيانات الحساسة الخاصة بالتنانت.
- يمكنه رؤية مؤشرات تشغيلية/تحليلية مجمعة.
- يمكنه التحكم في:
  - تمكين/تعطيل خدمات.
  - سياسات المنصة.
  - Templates وBlueprints.
  - Integrations العامة.
- دون الوصول لمحتوى العملاء الخاص.

## 12) الممنوعات الهندسية (Non-Negotiable Engineering Constraints)

### 12.1 ممنوعات أمنية

- ممنوع أي وصول مباشر لبيانات شركة أخرى.
- ممنوع أي bypass للـ RLS.
- ممنوع استخدام service role key في الـ client.
- ممنوع تمرير بيانات حساسة للـ AI بدون ضوابط.

### 12.2 ممنوعات معمارية

- ممنوع hard-coded text في الواجهات.
- ممنوع منطق صلاحيات داخل الواجهة فقط (يجب أن يكون Backend + RLS).
- ممنوع ربط التسعير بصفحة ثابتة قبل البروفيجنينج.
- ممنوع تفعيل Integrations مدفوعة دون صلاحية ودون تسجيل Audit.

### 12.3 ممنوعات تجربة المستخدم

- ممنوع عدم اتساق اللغة.
- ممنوع واجهات مزدحمة بالألوان.
- ممنوع إظهار ميزات غير مفعلة دون توضيح.
- ممنوع خلط صلاحيات الأدوار في نفس الشاشة.

## 13) معايير القبول (Acceptance Criteria) لأي جزء جديد

أي صفحة / موديول / خدمة جديدة لا تعتبر مقبولة إلا إذا حققت الآتي:

- أزرار وإشارات واضحة.
